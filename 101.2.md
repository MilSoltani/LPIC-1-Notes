## Bootloader
- BIOS/UEFI lädt Bootloader
  - Bootloader lädt Kernel
    - Kernel setzt Bootprozess fort
      - Bootprozess identifiziert und konfiguriert Hardware
      - ruft Dienstprogramm auf -> Start und die Verwaltung der Systemdienste
- Der Bootloader übergibt Parameter an den Kernel, z.B:
  - welche Partition das Root-Dateisystem enthält
  - in welchem Modus das Betriebssystem ausgeführt werden soll

## BIOS oder UEFI
### BIOS
- in einem nichtflüchtigen Speicherchip auf dem Mainboard gespeichert
- jedes Mal wenn der Computer eingeschaltet wird, wird ausgeführt
- Diese Art von Programm wird **Firmware** genannt
- Das BIOS lädt beim Start die ersten 440 Bytes des ersten Boot-Geräts als **ersten Bootloader**.
- Diese Bytes enthalten den sogenannten **Bootstrap-Code** (erste Startstufe).
- Die ersten 512 Bytes eines Laufwerks heißen **Master Boot Record (MBR)**.
- Der MBR enthält **Bootloader-Code** und zusätzlich die **Partitionstabelle**.
- Wenn der MBR fehlerhaft ist, kann das System nicht korrekt booten.
- BIOS Bootprozess – Vorbereitungsschritte
  - POST (Power-on Self-Test) prüft beim Start grundlegende Hardwarefehler.
  - BIOS initialisiert Basisgeräte wie Bildschirm, Tastatur und Speicherlaufwerke.
  - BIOS lädt die erste Bootloader-Stufe aus dem MBR (erste 440 Bytes des Startlaufwerks).
  - Erste Bootloader-Stufe startet die zweite Bootloader-Stufe -> verantwortlich für die Präsentation von Bootoptionen und das Laden des Kernels

### UEFI
- Eine Firmware wie BIOS, aber moderner aufgebaut.
- Kann Partitionen erkennen und verschiedene Dateisysteme lesen.
- Ist nicht auf den MBR angewiesen.
- Nutzt Einstellungen aus dem nichtflüchtigen Speicher (NVRAM) auf dem Mainboard.
  - Im NVRAM sind Speicherorte von EFI-Anwendungen gespeichert.
- EFI-Anwendungen können Bootloader, Diagnose- oder Reparaturtools sein.
- EFI-Programme liegen auf Partitionen in unterstützten Dateisystemen.
- Unterstützte Dateisysteme: FAT12, FAT16, FAT32 (Blockgeräte), ISO-9660 (optische Medien).
- Ermöglicht komplexere und flexiblere Startprozesse als BIOS.

- EFI-System-Partition (ESP) enthält EFI-Anwendungen.
- ESP darf nicht mit Root- oder Benutzerdatenpartitionen geteilt werden.
- EFI-Verzeichnis in der ESP enthält Programme, auf die NVRAM-Einträge verweisen.

- UEFI Bootschritte
  - POST prüft Hardware beim Systemstart.
  - UEFI initialisiert Basisgeräte wie Anzeige, Tastatur und Speicher.
  - UEFI liest NVRAM-Einträge und startet die definierte EFI-Anwendung aus der ESP.
  - Meist ist diese EFI-Anwendung ein Bootloader.
  - Bootloader lädt den Kernel und startet das Betriebssystem.

- Secure Boot
  - Erlaubt nur signierte EFI-Anwendungen.
  - Signaturen müssen vom Hardwarehersteller oder vertrauenswürdigen Stellen stammen.
  - Erhöht Schutz vor Malware im Bootprozess.
  - Kann die Installation nicht signierter Betriebssysteme einschränken.

## Der Bootloader
- GRUB (Grand Unified Bootloader) ist der meistgenutzte Bootloader für Linux auf x86-Systemen.
- Wird entweder vom BIOS oder von UEFI gestartet.

- Menüfunktion
  - Zeigt eine Liste verfügbarer Betriebssysteme.
  - Kann automatisch erscheinen oder manuell aufgerufen werden:
    - BIOS: Shift drücken
    - UEFI: Esc drücken
  - Kernel-Auswahl
    - Ermöglicht Auswahl verschiedener installierter Linux-Kernel.
    - Unterstützt das Übergeben von Kernel-Parametern.
  - Kernel-Parameter
    - Format: `option=value`

- Wichtige Linux Kernelparameter
  - **acpi**
    - Aktiviert/deaktiviert ACPI
    - Beispiel: `acpi=off` deaktiviert ACPI
  
  - **init**
    - Setzt alternativen Init-Prozess
    - Beispiel: `init=/bin/bash` startet direkt eine Shell
  
  - **systemd.unit**
    - Definiert systemd-Target
    - Beispiel: `systemd.unit=graphical.target`
    - Runlevels möglich (z. B. `1` oder `S` für Single-User)
  
  - **mem**
    - Begrenzt RAM-Nutzung
    - Beispiel: `mem=512M`
  
  - **maxcpus**
    - Begrenzung sichtbarer CPU-Kerne
    - Beispiel: `maxcpus=2`
    - `maxcpus=0` entspricht `nosmp`
  
  - **quiet**
    - Reduziert Boot-Ausgaben
  
  - **vga**
    - Setzt Grafikmodus
    - Beispiel: `vga=ask`
  
  - **root**
    - Definiert Root-Partition
    - Beispiel: `root=/dev/sda3`
  
  - **rootflags**
    - Mount-Optionen für Root-Dateisystem
  
  - **ro**
    - Root-Dateisystem nur lesbar beim Start
  
  - **rw**
    - Root-Dateisystem im Schreibmodus beim Start

- Konfiguration und Persistenz
  - Kernelparameter werden in `/etc/default/grub` gesetzt:
    - `GRUB_CMDLINE_LINUX`
  - Änderungen erfordern Regeneration der GRUB-Konfiguration:
    - `grub-mkconfig -o /boot/grub/grub.cfg`
  - Aktive Kernelparameter anzeigen:
    - `/proc/cmdline`

## Systeminitialisierung
- Kernel lädt nach dem Bootloader weitere Systemkomponenten  
  - Shell-Skripte für kurze Initialisierungsaufgaben  
  - Daemons (Dienste) für dauerhaft laufende Systemfunktionen  

- Startskripte vs. Daemons  
  - Skripte: einmalige Aufgaben während des Starts  
  - Daemons: laufen dauerhaft im Hintergrund  
  - Dienste können gestartet, gestoppt und neu gestartet werden  
  - Änderungen an Konfiguration erfordern oft einen Neustart des Dienstes  

- Systemverwaltung  
  - Administrator kann gezielt Dienste aktivieren oder deaktivieren  
  - Minimalbetrieb möglich (nur notwendige Dienste aktiv)  
  - Wird oft für Wartung oder Sicherheit genutzt  

- Kernelstartprozess  
  - Bootloader lädt Kernel in den RAM  
  - Kernel übernimmt CPU-Steuerung  
  - Hardwareerkennung und Speicherverwaltung werden initialisiert  

- initramfs  
  - temporäres Root-Dateisystem im RAM  
  - enthält Module für Zugriff auf echtes Root-Dateisystem  
  - wird nach erfolgreichem Start entfernt  

- Übergang zum echten System  
  - Kernel bindet Dateisysteme aus `/etc/fstab` ein  
  - startet erstes User-Space-Programm `init`  
  - `init` startet Dienste und Initialisierungsskripte  

- System-Init-Varianten  
  - SysV init  
    - nutzt Runlevel (0–6)  
    - Runlevel 0, 1 und 6 sind standardisiert  
  - systemd  
    - moderner Service-Manager  
    - nutzt Sockets, D-Bus und cgroups  
    - unterstützt On-Demand-Dienste und Abhängigkeitsmanagement  
    - weit verbreiteter Standard in modernen Distributionen  
  - Upstart  
    - optimiert Boot durch paralleles Starten von Diensten  
    - früher in Ubuntu genutzt  
    - heute weitgehend durch systemd ersetzt  

## Initialisierungsprüfung
- Während des Bootvorgangs können nicht-kritische Fehler auftreten  
  - System startet trotzdem weiter  
  - Verhalten kann dadurch beeinflusst werden  

- Alle Fehler erzeugen Meldungen  
  - enthalten Informationen über Zeitpunkt und Ursache  
  - hilfreich für Analyse und Debugging  

- Auch ohne sichtbare Ausgabe werden Boot-Informationen gespeichert  
  - werden für spätere Diagnose genutzt  
  - können für Konfiguration und Optimierung relevant sein  

- Kernel-Ringpuffer  
  - Speicherbereich für Kernelmeldungen  
  - enthält auch Bootmeldungen  
  - bleibt aktiv auch wenn Meldungen nicht angezeigt werden  
  - kann durch Animationen oder Splash-Screens verdeckt sein  

- Verhalten des Ringpuffers  
  - Inhalte gehen beim Herunterfahren verloren  
  - kann manuell geleert werden mit `dmesg --clear`

- Die Ausgabe von `dmesg` kann sehr umfangreich sein  
  - oft hunderte Zeilen  
  - zeigt Kernel-Initialisierung und Systemstartprozesse  
  - Beispielinhalt: Start von systemd und Treiberinitialisierung  

- Zeitstempel in `dmesg`  
  - Werte zeigen Sekunden seit Kernelstart  
  - dienen zur zeitlichen Einordnung von Ereignissen  

- journalctl (systemd-Systeme)  
  - zeigt Initialisierungs- und Systemmeldungen  
  - ersetzt bzw. ergänzt `dmesg` in modernen Systemen  

- Wichtige Optionen von `journalctl`  
  - `-b` oder `--boot`: aktuelle Boot-Meldungen  
  - `-k` oder `--dmesg`: Kernel-Meldungen  
  - `--boot`: gleiche Funktion wie `-b`  

- Boot-Übersicht  
  - `journalctl --list-boots`  
  - zeigt alle gespeicherten Boots  
  - enthält Boot-ID (Hash)  
  - zeigt Start- und Endzeit jedes Boots

  - Logs aus vorherigen Systemstarts bleiben verfügbar  
  - ermöglichen Analyse vergangener Bootvorgänge  

- Boot-Auswahl in `journalctl`  
  - `-b 0` oder `--boot=0`: aktueller Boot  
  - `-b -1` oder `--boot=-1`: vorheriger Boot  
  - `-b -2` oder `--boot=-2`: Boot davor  
  - negative Werte zählen rückwärts durch frühere Starts  

- Zweck der Boot-Historie  
  - Vergleich von Systemstarts  
  - Fehleranalyse über mehrere Reboots hinweg  
  - Nachverfolgung von Kernel- und Serviceproblemen

- Systemmeldungen und Initialisierungslogs werden in `/var/log/` gespeichert  
  - enthält Protokolle zu Boot, Diensten und Fehlern  
  - hilfreich zur Analyse von Systemproblemen  

- Bei schwerwiegenden Bootfehlern  
  - System startet ggf. nicht vollständig  
  - Lösung: Boot von alternativem Medium (Live-System)  
  - Zugriff auf internes Dateisystem möglich  
  - Logs in `/var/log/` werden zur Fehleranalyse genutzt  

- systemd Journal  
  - speichert Logs nicht als reinen Text  
  - benötigt `journalctl` zum Auslesen  

- journalctl Speicherorte  
  - Standard: `/var/log/journal/`  
  - kann alternative Verzeichnisse lesen  

- journalctl Optionen  
  - `-D` oder `--directory`  
    - liest Journal-Daten aus einem anderen Verzeichnis  
    - z. B. von einem gemounteten System
